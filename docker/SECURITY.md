# Security model — Pyongyang Racer play server

This document describes what is exposed, what is protected, and recommended hardening.

## What players connect to

| Layer | Exposed? | Notes |
|-------|----------|-------|
| **GCP/Oracle VM (host)** | No (via game URL) | Only port **6080** is opened for the game. SSH (22) is separate and key-only. |
| **Docker container** | Yes | noVNC streams the Flash Player window over HTTPS. |
| **Full VM desktop** | No | There is no window manager and no terminal in the container. VNC is clipped to the Flash window only. |

Closing the Flash window shows a black screen until supervisor restarts the game. Players cannot browse the host VM filesystem or run shell commands through noVNC.

## Authentication

- **VNC password** is required (stored in `docker/.vnc-password`, auto-generated on first deploy).
- **Default `pyongyang` is no longer used** for new deployments.
- x11vnc listens on `localhost:5900` inside the container — only the noVNC proxy can reach it, not the public internet directly.
- noVNC is served over **HTTPS** with a self-signed certificate (browser warning is expected).

### Residual risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Weak/guessable password | High | Auto-generated 20-char password; set `VNC_PASSWORD` yourself for production |
| Password in URL query string | Medium | URL may appear in browser history; prefer typing password in the noVNC prompt for shared PCs |
| Port 6080 open to world | Medium | Restrict GCP/Oracle firewall to your IP: `YOUR_IP/32` |
| Old Flash runtime in container | Medium | Container has no browser, no shell, no outbound tools; Flash only loads the bundled SWF |
| Docker escape → host | Low | Standard container isolation; keep Docker updated on the VM |
| Brute-force VNC password | Low | Use long password + IP allowlist |

## Container hardening (docker-compose.yml)

- `cap_drop: ALL` — no Linux capabilities
- `no-new-privileges:true` — blocks privilege escalation
- No shell, terminal, or window manager in the container
- VNC streams **only the Flash window** (not a full desktop)
- Game files are baked into the image; Flash config writes go to `/tmp` only

## Game fidelity

The server runs the **same files** as the Mac app:

- `PY Racer_0509/PYracer.swf` (unchanged SWF)
- Same `.dat` / `.txt` / `.mp3` assets
- Adobe Flash Player **standalone** (Linux build of the same SA family as Mac)

Controls, physics, and animations are entirely inside the original SWF — nothing in this stack modifies game logic.

## Performance limits (not security)

The free **e2-micro** VM (1 shared vCPU, 1 GB RAM) plus browser VNC streaming cannot match native Mac performance. See `docker/PERFORMANCE.md`.

## Recommended production checklist

1. [ ] Restrict firewall: TCP 6080 from `YOUR_IP/32` only
2. [ ] Use a strong `VNC_PASSWORD` (auto-generated by `./oracle/deploy.sh`)
3. [ ] Keep VM and Docker updated: `sudo apt-get update && sudo apt-get upgrade`
4. [ ] Do not commit `docker/.vnc-password` to git (already in `.gitignore`)
5. [ ] Rotate password if you suspect it was leaked: `VNC_PASSWORD='new-secret' ./oracle/deploy.sh`

## What we cannot guarantee

- **Adobe Flash Player on Linux** is unmaintained and may crash (supervisor auto-restarts it).
- **Segfaults** in Flash are upstream binary bugs — we cannot patch them; we isolate them inside the container.
- **100% uptime** on free-tier VMs is not guaranteed by Google or Oracle.
